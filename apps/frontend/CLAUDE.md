## Map rendering â€” inline SVG, no map library

Per the root CLAUDE.md Â§5, the map is **inline SVG** â€” no tile-based map library, no
basemap, no API key. `src/viz/projection.ts` projects real IBGE GeoJSON to SVG paths once
at load using a small custom equirectangular projection (with cos-latitude aspect
correction) rather than `d3-geo`; pan/zoom and choropleth scales are likewise hand-rolled
(`src/viz/modes.ts`) to keep the dependency footprint minimal. The result works fully
offline and gives crisp, highlightable, freely-zoomable state borders â€” the Victoria 3 /
EU-style border highlighting the brief asked for. Geometry is real IBGE data; only the
indicator values are synthetic (see below).

## Data â€” live API, with synthetic fallback

`src/data/dataset.ts` is the data layer and **the one swap point**. On load it calls the
NestJS API (`VITE_API_URL`, default `http://localhost:3000`), discovers the country/level/
periods from `/countries`, fetches `/countries/:code/regions`, and **adapts the API's
nested-by-theme shape into the flat `StateRecord`** the modes consume (renaming
`pib_per_capita`â†’`gdp_per_capita`, `gdp_share_*`â†’`sector_*`, converting GDP to millions,
deriving `dominant_sector`). `BR_DATA.source` is `"live"` or `"synthetic"`.

If the API is unreachable it falls back to `src/data/synthetic.ts` (seeded illustrative
values for 2010/2016/2022) so the UI still runs standalone; `BR_DATA.source` records
whether `"live"` or `"synthetic"` is active (no UI badge).

**Coverage is partial by design.** The IBGE worker feeds Demographics + Economic Profile
today; the other six modes are **greyed out** (`isModeAvailable` in `viz/modes.ts`) until
their source workers (SICONFI/DataSUS/ANEEL/TransparĂŞncia) land. Missing indicator rows
render "â€”".

**Geometry stays bundled.** Real borders are deferred (root Â§10 geometry decision), so the
API serves no geometry; the map keeps rendering `src/data/br-states.geo.ts`. Real,
non-synthetic reference data: `states-meta.ts` (names/capitals/codes/areas) + the geometry.

**Map layers (mutually exclusive).** Beyond the default state (UF) view there are two extra
layers behind topbar toggles â€” **municipalities** (N6) and **macro-regions** (N2) â€” held in a
single `layer: "uf" | "municipio" | "regiao"` enum in `App.tsx` (persisted `bv.layer`), so
they're mutually exclusive by construction. The toggle button is the generic `LayerToggle`
(`controls.tsx`); both are data-driven per mode the same way (own-value choropleth where the
mode has data, else fall back) and share the lazy `fetchLevel()` loader in `dataset.ts`.

**Municipality layer (data-driven).** Overlays all 5,570 municipalities. Two lazy chunks
load on first enable (initial load unaffected): the **mesh**
(`src/data/br-municipalities.geo.ts`, ~2.8MB real IBGE borders, generated by
`scripts/build-municipalities-geo.mjs`, imported via `src/data/municipalities.ts`) and the
**indicator data** (`loadMuniData()` in `dataset.ts`, fetched from `/regions?level=municipio`).
- **Coloring is data-driven per mode.** When the active mode has N6 data (Demographics,
  Economic Profile), each municipality is colored by its **own** value on a
  municipality-derived scale (`makeScale(..., { robust: true })` â€” clamps to the 2ndâ€“98th
  percentile so outliers don't wash out the map). Otherwise (Fiscal/Energy â€” state-only
  sources) it falls back to its **parent state's** color (`code.slice(0,2)` â†’ state record).
- **Interaction follows the data.** In own-data modes the layer is interactive: hover â†’
  `MuniTooltip` (name + active metric), click â†’ selects the **parent state** (opens the rich
  state detail; municipalities have no metadata/trend of their own). In fallback modes it's
  non-interactive (`pointer-events: none`) and the state layer beneath handles hover.
- The `MunicipalityLayer` is `React.memo`'d so it recolors only on mode/year/palette change,
  never on hover/pan/zoom. The legend reflects the municipality scale when it's active.
Regenerate the mesh with `node scripts/build-municipalities-geo.mjs`.

**Macro-region layer (data-driven).** The 5 dissolved IBGE macro-regions (Norteâ€¦Centro-Oeste).
Its geometry is **tiny (~41KB) so it's bundled eagerly** (`src/data/br-regions.geo.ts`,
generated by `scripts/build-regions-geo.mjs`, projected in `projection.ts` â†’ `BR_REGION_GEO`);
only the 5 indicator rows lazy-load (`loadRegionData()`, `/regions?level=regiao`). `RegionLayer`
colors each region by its own value; hover â†’ `PlaceTooltip` (shared with municipalities). A
region has no parent, so **click is a no-op**. For modes without N2 data (Fiscal/Energy) the
layer isn't drawn and the state choropleth shows through. A state's region = first digit of its
UF code. Regenerate with `node scripts/build-regions-geo.mjs`.

## Conventions

- **Indicator/theme/mode keys are data, not i18n keys** â€” translated under the
  `ind.` / `theme.` / `mode.` / `cat.` namespaces. Missing keys fall back to the raw key.
- Add every new user-facing string to **both** `en` and `pt-BR` in `i18n/strings.ts`.
- The mode/scale/palette system in `viz/modes.ts` is the one place choropleth logic
  lives â€” sequential, diverging (with a meaningful midpoint), and categorical scales.
- **Metric override:** clicking an indicator row in the detail panel recolors the whole map
  by that metric. `App` holds `metricProp`; when set it derives an "effective mode"
  (sequential scale on that prop) passed to the map/legend/tooltip/overview. Clicking the
  active row toggles back to the mode default; switching modes clears it. Rows for indicators
  with no data are non-clickable.
- User prefs (lang, mode, year, tweaks) persist to `localStorage` under `bv.*`.
